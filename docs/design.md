# Normalisation Rules Webapp — Design & Implementation Spec

## Overview

A Flask + HTMX + Alpine.js single-page webapp for managing attribute normalisation
rules that map raw sample metadata values to canonical structured values. Rules apply
as a **post-processing step** on top of the crawlers' existing extraction logic.

---

## Background

The crawlers (`geo.py`, `arrayexpress.py`) extract metadata from GEO and ArrayExpress
into structured columns (`tissue`, `disease`, `gender`, `age`, `extraction_protocol`)
and an unstructured `extras` JSONB column. Raw values are highly heterogeneous — e.g.
`"whole blood"`, `"Whole Blood"`, `"PBMC"`, `"peripheral blood mononuclear cells"` all
represent blood-derived tissue, but currently land verbatim in the DB.

Normalisation rules allow a curator to define patterns (substring, glob, regex) that
match raw values and rewrite them to canonical terms. Rules are **not auto-applied on
save** — a curator clicks **Apply All Rules** to batch-update samples.

## Schema Migration

The existing `normalisation_rule` table has no primary key and all nullable columns.
Apply this migration before running the webapp:

```sql
alter table normalisation_rule
    add column id               bigint generated by default as identity primary key,
    add column source_attribute text not null,
    add column priority         integer not null default 0,
    add column created_at       timestamptz not null default now(),
    alter column pattern          set not null,
    alter column rule_type        set not null,
    alter column target_attribute set not null,
    alter column attribute_value  set not null;
```

### Column semantics

| Column | Description |
|---|---|
| `id` | Surrogate primary key |
| `source_attribute` | Where the pattern is matched (see below) |
| `pattern` | The text pattern to match against |
| `rule_type` | `substring` \| `glob` \| `regex` |
| `target_attribute` | Which column to write the result into |
| `attribute_value` | The canonical value to write on match |
| `priority` | Higher value = evaluated first; first match wins |
| `created_at` | Audit timestamp |

### `source_attribute` semantics

- If the value is one of `{tissue, disease, gender, age, extraction_protocol}`:
  match against that **structured column** in `sample`.
- Otherwise: match against `sample.extras->>'<source_attribute>'`
  (a key in the `extras` JSONB column).

This lets rules normalise both structured fields (e.g. tidy up messy tissue strings
already extracted by the crawler) and unstructured extras keys (e.g. `"cell type"`
or `"treatment"` keys that landed in `extras`).

## API Routes

| Method | Path | Description | Returns |
|---|---|---|---|
| `GET` | `/` | Main page | `index.html` full page |
| `GET` | `/rules` | List rules; optional `?target=<attr>` filter | `rule_list` partial |
| `POST` | `/rules` | Create a rule | updated `rule_list` partial |
| `PUT` | `/rules/<id>` | Update a rule | updated `rule_row` partial |
| `DELETE` | `/rules/<id>` | Delete a rule | empty string (HTMX removes row) |
| `GET` | `/preview` | Preview `?source=<attr>&limit=50` | `preview_table` partial |
| `POST` | `/apply` | Apply all rules to all samples | JSON `{"updated": N}` |

Pure functions with no database I/O — fully unit-testable.

```python
    """
```

### `apply_rules_to_sample` — field lookup

For each distinct `source_attribute` in `rules`:
- If `source_attribute` in `STRUCTURED_FIELDS` → read `sample[source_attribute]`
- Else → read `sample.get('extras', {}).get(source_attribute)`

If the raw value is `None` / empty string, no rules are evaluated for that field.

## DB-Level Apply (`POST /apply` in server.py)

```python
def apply_all_rules(conn):
    rules = fetch_all_rules_sorted(conn)        # ORDER BY priority DESC
    samples = fetch_all_samples(conn)           # id + all structured cols + extras
    updated = 0
    for sample in samples:
        changes = apply_rules_to_sample(sample, rules)
        if changes:
            update_sample(conn, sample['id'], changes)
            updated += 1
    return updated
```

**Gender cast**: When `target_attribute == 'gender'`, the value must be cast to the
`gender` postgres enum (`'male'`, `'female'`, `'unknown'`). Use a parameterised query:

```sql
UPDATE sample SET gender = %s::gender WHERE id = %s
```

For other structured fields (text columns), a plain `%s` parameter suffices.

For **extras rules** (where `target_attribute` is not a structured field):
```sql
UPDATE sample
   SET extras = jsonb_set(coalesce(extras, '{}'), '{target_attribute}', '"value"')
 WHERE id = %s
```

## Template Structure

### `base.html`
Standard HTML5 skeleton. Loads HTMX 2.x and Alpine.js 3.x from CDN.
Defines a `{% block content %}` slot and a `{% block scripts %}` slot.

### `index.html`
Extends `base.html`. Two-column layout (CSS Grid or Flexbox).

Left column:
- Filter dropdown (`?target=<attr>`) — `hx-get="/rules"`, `hx-trigger="change"`, `hx-target="#rule-list"`
- `<div id="rule-list">` — initial render calls `{% include 'partials/rule_list.html' %}`
- Add Rule button — reveals `rule_form.html` partial inline or in a modal

Right column:
- Source attribute dropdown — `hx-get="/preview"`, `hx-trigger="change"`, `hx-target="#preview-table"`
- `<div id="preview-table">` — initial render calls `{% include 'partials/preview_table.html' %}`

Apply All Rules button:
```html
<div x-data="{ loading: false, result: null }">
  <button
    @click="loading = true; result = null;
            fetch('/apply', {method: 'POST'})
              .then(r => r.json())
              .then(d => { result = d.updated; loading = false; })"
    :disabled="loading"
  >
    <span x-show="!loading">Apply All Rules</span>
    <span x-show="loading">Applying…</span>
  </button>
  <span x-show="result !== null" x-text="`Updated ${result} samples`"></span>
</div>
```

### `partials/rule_list.html`
Iterates over `rules` and includes `rule_row.html` for each. Also includes
`rule_form.html` at the bottom for the "Add Rule" form.

### `partials/rule_row.html`
Displays one rule in read-only mode:
- Columns: source_attribute → target_attribute, pattern, rule_type, priority
- Edit button: sets `editing = true` (Alpine) or swaps in `rule_row_edit.html`
- Delete button: `hx-delete="/rules/{{ rule.id }}"`, `hx-target="#rule-list"`, `hx-confirm="Delete this rule?"`

### `partials/rule_row_edit.html`
Inline edit form for an existing rule:
- Pre-filled `<input>` / `<select>` fields
- Save: `hx-put="/rules/{{ rule.id }}"`, `hx-target="#rule-list"`
- Cancel: sets `editing = false` (Alpine)

### `partials/rule_form.html`
Form for creating a new rule:
- Fields: source_attribute, pattern, rule_type (select), target_attribute, attribute_value, priority
- Submit: `hx-post="/rules"`, `hx-target="#rule-list"`

### `partials/preview_table.html`
Table with columns: Sample ID, Raw Value, Matched Rule ID, Result.
- Rows come from the `/preview?source=<attr>&limit=50` endpoint.
- "—" shown when no rule matches.
- Matched rule ID links to the rule row for quick navigation.

## Testing Strategy

### Unit tests (`test_normalise.py`)

Cover `match_value` for all three rule types, case-insensitivity, edge cases
(empty string, None-safe behaviour). Cover `first_matching_rule` for priority
ordering and no-match case. Cover `apply_rules_to_sample` for structured field
lookup, extras lookup, and gender value pass-through.

No DB or HTTP fixtures needed — all inputs are plain dicts.

### Integration test (manual)

1. Start Flask dev server.
2. Navigate to `http://localhost:5000`.
3. Add a rule: source=`tissue`, pattern=`pbmc`, type=`substring`, target=`tissue`, value=`blood`.
4. Check Preview panel shows matching samples.
5. Click Apply All Rules — verify toast shows updated count.
6. Query DB: `SELECT tissue FROM sample WHERE tissue = 'blood' LIMIT 5;`
